Lesson 4 — Common beginner mistakes

Estimated time: 40 min · Bloom focus: Analyze

1. What you'll learn

The handful of mistakes that trip up almost every beginner — and how to spot them early.

2. Why it matters

Knowing the failure modes is half of competence. Each mistake below has cost someone hours.

3. Simple explanation — the top five

  1. Relying on latest. It's the default tag, not "newest stable". Builds become non-reproducible. → Pin versions: python:3.11-slim.
  2. Losing data. No volume → data gone on rm. → Mount a volume for anything you care about.
  3. Forgetting -p. The service runs but is unreachable from the browser. → Publish the port.
  4. Huge images. Using a full OS base and copying everything (including node_modules, .git). → Use slim bases and a .dockerignore.
  5. Putting secrets in the image. Baking passwords into a Dockerfile leaks them to anyone with the image. → Pass via -e / secrets at runtime.

4. Concrete example

A 1.2 GB image built FROM ubuntu with everything copied in. Switching to python:3.11-slim + a .dockerignore dropped it to ~120 MB and builds got faster thanks to layer caching.

5. Correct technical view

Order Dockerfile instructions from least- to most-frequently-changing so the layer cache stays valid: copy requirements.txt and install deps before copying source. Then editing source doesn't bust the dependency layer.

6. Common mistakes (meta)

  • Debugging blind. Use docker logs <id> and docker exec -it <id> sh to look inside a running container.
  • Running everything as root inside the container without thinking about it.

7. Practical case

"My API container exits immediately." docker logs shows it crashed on a missing env var — fixed with -e. Without reading logs, this looks like "Docker is broken".

8. Recall questions

Why is pinning an image tag better than using latest?

It makes builds reproducible; latest can change under you. latest is slower to download.

Where should secrets like passwords go?

Provided at runtime (env vars / secrets), never baked into the image. Hard-coded in the Dockerfile for convenience.

What command lets you inspect why a container crashed?

docker logs <id> docker images

9. Mini-exercise

Take the Dockerfile from Lesson 2 and list two changes that would (a) shrink the image and (b) improve layer-cache reuse.

10. Summary

The classic five: latest reliance, data loss without volumes, missing -p, bloated images, and baked-in secrets. Order Dockerfile steps to preserve the cache. When something breaks, read docker logs and exec into the container instead of guessing.

11. Terms for the glossary

.dockerignore, layer cache, slim image, docker logs, docker exec.
