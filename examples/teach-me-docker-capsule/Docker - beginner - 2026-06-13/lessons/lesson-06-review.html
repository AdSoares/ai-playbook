Lesson 6 — Review & consolidation

3. Simple explanation — the whole picture in one paragraph

A Dockerfile is a recipe; docker build turns it into a read-only image (made of cached layers); docker run starts a container (a running instance with a throwaway writable layer). You publish ports with -p, persist data with -v volumes, and share images via a registry. Containers share the host kernel, so they're light and fast.

4. Concrete example — command cheat sheet

build: docker build -t name . run: docker run -d -p 8080:80 -v data:/path -e KEY=val name list: docker ps docker images debug: docker logs <id> docker exec -it <id> sh clean: docker stop <id> docker rm <id>

5. Correct technical view — self-explanation prompt

Without looking, explain aloud: (a) image vs. container, (b) why data is ephemeral, (c) what -p 8080:80 does, (d) why latest is risky. If any is shaky, revisit that lesson — that's metacognition working for you.

6. Common mistakes recap

latest reliance · no volume → lost data · missing -p · bloated images · baked-in secrets.

8. Recall questions

One image, run three times — how many containers?

Three containers from one image. One container reused three times.

Which flag persists data across container removal?

-v (a volume mount). -d (detached mode).

Containers are lightweight compared to VMs mainly because…

they share the host kernel instead of bundling a full guest OS. they compress the application binary.
