Lesson 2 — Images, containers, registries & the Dockerfile

Estimated time: 1h10 · Bloom focus: Understand, Apply

1. What you'll learn

The four core nouns of Docker and how they relate: image, container, registry, and Dockerfile.

2. Why it matters

Every Docker command you'll ever type manipulates one of these. Get the mental model right and the CLI stops feeling like magic.

3. Simple explanation

  • Image — a frozen, read-only template (your app + environment). Like a class.
  • Container — a running instance created from an image. Like an object.
  • Registry — a store of images you can pull from / push to (e.g. Docker Hub).
  • Dockerfile — a recipe (text file) describing how to build an image.

4. Concrete example

You docker pull nginx (image from a registry), then docker run nginx starts a container from it. Run it three times → one image, three containers, like three objects from one class.

5. Correct technical view

An image is built from layers — each instruction in a Dockerfile adds a layer, cached and reused across builds. A container adds a thin writable layer on top of the image's read-only layers; that writable layer is discarded when the container is removed (this is why container data is ephemeral — Lesson 3). Images are identified by name and tag (e.g. nginx:1.27); the tag defaults to latest if omitted.

# Minimal Dockerfile FROM python:3.11-slim # base image (a layer) WORKDIR /app # set working dir COPY app.py . # copy your code in (a layer) CMD ["python", "app.py"] # default command when a container starts

6. Common mistakes

  • Confusing image and container ("restart the image" — you restart a container).
  • Assuming latest is the newest stable version. It's just the default tag and can point anywhere; pin a real version.

7. Practical case

A teammate gives you Dockerfile + code. You docker build -t myapp . to produce the image myapp, then docker run myapp to get a container. The registry isn't even needed locally.

8. Recall questions

What is the relationship between an image and a container?

An image is an immutable template; a container is a running instance created from it. They are two words for the same thing.

What does a Dockerfile do?

It describes, step by step, how to build an image. It runs a container in the background.

What is nginx:1.27 — and what does 1.27 mean?

An image name with a tag; the tag selects a specific version/variant. A container ID assigned at runtime.

9. Mini-exercise

Write, from memory, a 4-line Dockerfile for a Python script named main.py. Then say which line creates the base layer and which line sets the default command.

10. Summary

Image = immutable template (built from cached layers). Container = running instance with a thin writable layer that's discarded on removal. Registry = where images live (Docker Hub). Dockerfile = the recipe to build an image. Images are name:tag; latest is just the default tag, not "newest stable". Build with docker build, run with docker run. This image↔container distinction is the single most tested idea at this level.

11. Terms for the glossary

image, container, registry, Docker Hub, Dockerfile, layer, tag, base image.
