Lesson 5 — Case study: containerize a small web app

Estimated time: 1h00 · Bloom focus: Apply, Create

1. What you'll learn

Put it together: take a tiny app from source to a running container you built yourself.

2. Why it matters

Building your own image is the moment Docker stops being abstract.

3. Simple explanation — the app

A minimal Python web server (app.py) that returns "Hello from a container":

# app.py from http.server import BaseHTTPRequestHandler, HTTPServer class H(BaseHTTPRequestHandler): def do_GET(self): self.send_response(200); self.end_headers() self.wfile.write(b"Hello from a container") HTTPServer(("0.0.0.0", 80), H).serve_forever()

4. Concrete example — the Dockerfile

FROM python:3.11-slim WORKDIR /app COPY app.py . EXPOSE 80 CMD ["python", "app.py"]

Note 0.0.0.0 in the app: binding to localhost inside the container would be unreachable from the host — a classic gotcha.

5. Correct technical view — build & run

docker build -t hello-web . # build image named hello-web from ./Dockerfile docker run -d -p 8080:80 hello-web # run it, publish to localhost:8080 # open http://localhost:8080 -> "Hello from a container" docker logs <id> # see server output docker stop <id> && docker rm <id> # clean up

The . in build is the build context — the folder sent to Docker. EXPOSE documents the port; -p actually publishes it.

6. Common mistakes

  • Binding to 127.0.0.1 inside the container → unreachable from host. Use 0.0.0.0.
  • Expecting EXPOSE alone to make the app reachable. You still need -p.

7. Practical case

You hand a colleague the folder (app.py + Dockerfile). They run two commands and see the same page — no Python setup instructions needed. That's the whole point of Lesson 1, realized.

8. Recall questions

Why bind the server to 0.0.0.0 instead of 127.0.0.1 inside a container?

So it accepts connections forwarded from the host; 127.0.0.1 would only accept from inside the container. 0.0.0.0 makes the server faster.

What does the . mean in docker build -t hello-web .?

The build context — the current folder sent to Docker (where the Dockerfile is). It tells Docker to hide the output.

Does EXPOSE 80 publish the port to your machine?

No — it only documents the port; you still need -p to publish it. Yes — EXPOSE alone makes localhost reach the app.

9. Mini-exercise

Change the response text, rebuild with a new tag (hello-web:v2), run it on port 9090, and confirm both v1 and v2 can run at once on different ports.

10. Summary

You wrote a Dockerfile, built an image with docker build -t, ran it with -p, and verified it in the browser. Key gotchas: bind to 0.0.0.0; EXPOSE documents while -p publishes; the . is the build context. You can now package a simple app end-to-end.

11. Terms for the glossary

build context, EXPOSE, CMD, 0.0.0.0 vs 127.0.0.1, image tagging.
